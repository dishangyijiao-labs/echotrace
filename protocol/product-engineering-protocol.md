# 产品工程协作协议

## 一、概述

### 适用范围

- 面向用户产品（App / Web / SaaS）
- 人 + AI 协作开发

### 核心原则

- 可验证
- 可结构化
- 风险显式化
- 安全强制检查

---

### 核心命令

/run proto-0   # 产品定义
/run proto-1   # 需求分析

## 三、协议定义

---

### 0. 产品定义协议

**目标：** 定义产品方向（做什么、为谁做、为什么做）

**输入：**
- 用户问题
- 数据分析
- 业务目标

**输出：**
- 问题陈述
- 目标用户
- 使用场景
- 北极星指标

**风险：**
- 伪需求
- 用户定位错误

**安全检查：**
- 是否涉及敏感数据或群体

**验收标准：**
- 明确回答：谁 / 为什么 / 成功标准

---

### 1. 需求分析协议

**目标：** 输出可执行的产品需求文档

**输入：**
- 产品定义

**输出（产品需求文档）：**
- 问题定义
- 用户定义
- 用户路径
- 功能需求
- 非功能需求
- 指标
- 范围
- 风险

**风险：**
- 需求歧义
- 范围膨胀

**安全检查：**
- 是否涉及个人隐私信息
- 权限是否明确
- 数据是否最小化

**验收标准：**
- 设计和开发可直接执行

### 10. 迭代协议

**目标：** 确保每次迭代方向正确，不做没人用的功能

**核心问题（每次迭代前必须回答）：**
1. 这个功能解决了谁的什么问题？
2. 这个问题是我假设的，还是有真实依据？
3. 如果做完了没人用，损失的时间是否可以接受？

**迭代来源优先级：**
1. 自己使用产品时发现的痛点
2. 真实用户的反馈
3. 安全和稳定性问题
4. 技术债务（影响开发效率时才处理）

**防止方向偏移的检查：**
- 每次决定做新功能前，问自己："如果这个功能做完了，3 个月后我还觉得它有价值吗？"
- 如果答案不确定，先做最小验证版本
- 不因为"竞品有"就做。问"我的用户需要吗"

**输入：**
- 自己使用中的发现
- 用户反馈
- 安全审计结果

**输出：**
- 下一轮迭代的需求，进入 proto-0 或 proto-1

**验收标准：**
- 能清楚说出"这次迭代为什么做这个"
- 做完后能验证"是否解决了那个问题"
